# MDXForge Extension Packages

MDXForge can evolve from a local MDX reader into a local extension host for AI-authored documents. The goal is to let users install and share component packages while keeping MDX files clean, portable, and predictable.

<Callout title="Core principle">
MDX documents should use registered components, not import arbitrary local code. Extensions own code loading; MDX stays a readable document protocol.
</Callout>

## Product Goal

The target experience is:

```txt
user installs a package
-> package registers MDX components and writing rules
-> AI can read those rules
-> MDX files use the registered components
-> MDXForge renders the document locally
```

For example:

```bash
mdxforge install npm:@acme/mdxforge-charts
mdxforge install git:https://github.com/acme/mdxforge-api-docs
mdxforge install ./local-extension
```

Then an MDX file can use package-provided components without import statements:

```mdx
---
title: Revenue Review
description: A local report rendered with an installed chart extension.
---

# Revenue Review

<MetricCard label="MRR" value="$24,800" trend="+12%" />

<RevenueChart
  data={[
    { month: "Jan", value: 12000 },
    { month: "Feb", value: 18000 }
  ]}
/>
```

This model is similar in spirit to Pi packages: users can install shareable packages from npm, git, or local paths, and a public catalog can discover packages by metadata. MDXForge should adapt that idea to document rendering instead of agent automation.

## Non-Goals

The extension system should not turn every MDX file into a free-form React app.

- Do not require MDX documents to write `import` or `export`.
- Do not let normal component packages access Node APIs by default.
- Do not give renderer components direct `fs`, `shell`, or process access.
- Do not make arbitrary TSX compilation part of every document render.
- Do not block the first version on a full plugin marketplace.

## Package Types

MDXForge packages should be typed by capability. This makes permissions, review, and user expectations clearer.

| Type | Purpose | Default Permission |
| --- | --- | --- |
| `component` | Register MDX components. | Low |
| `theme` | Add reading themes and CSS tokens. | Low |
| `rules` | Add AI writing rules only. | Low |
| `template` | Add starter MDX documents or workspace templates. | Low |
| `transform` | Add remark/rehype compile plugins. | High |
| `resource` | Customize local asset resolution. | High |

The first implementation should focus on `component`, `theme`, `rules`, and `template`. `transform` and `resource` extensions are more powerful because they touch compile behavior or local files, so they should require explicit trust prompts.

## Package Shape

A package should be a normal npm-compatible package with MDXForge metadata in `package.json`.

```json
{
  "name": "@acme/mdxforge-charts",
  "version": "1.0.0",
  "description": "Chart and metric components for MDXForge documents.",
  "keywords": ["mdxforge-package", "mdxforge-extension"],
  "peerDependencies": {
    "react": "^19.0.0",
    "react-dom": "^19.0.0"
  },
  "mdxforge": {
    "type": "component",
    "entry": "./dist/extension.js",
    "styles": ["./dist/style.css"],
    "rules": ["./rules.mdx"],
    "preview": {
      "image": "./preview.png",
      "demo": "./demo.mdx"
    }
  }
}
```

The entry file exports a declarative extension object:

```ts
import { RevenueChart } from './components/RevenueChart'
import { MetricCard } from './components/MetricCard'

export default {
  name: '@acme/mdxforge-charts',
  version: '1.0.0',
  components: {
    RevenueChart,
    MetricCard
  },
  componentRules: [
    {
      name: 'MetricCard',
      description: 'Display one KPI with a label, value, and optional trend.',
      props: {
        label: { type: 'string', required: true },
        value: { type: 'string', required: true },
        trend: { type: 'string' }
      },
      examples: ['<MetricCard label="MRR" value="$24,800" trend="+12%" />']
    }
  ]
}
```

The package can also ship `rules.mdx`, which explains how AI should use its components.

```mdx
---
title: Charts Extension Rules
description: Writing rules for @acme/mdxforge-charts.
---

# Charts Extension Rules

Use `<MetricCard />` for single KPIs. Use `<RevenueChart />` only when the data is a small static array.

Never fetch data from the network inside MDX. Put data directly in props or link to a local source file.
```

## Extension API

MDXForge should expose a small helper package for extension authors:

```ts
import { defineMdxForgeExtension } from '@mdxforge/extension'

export default defineMdxForgeExtension({
  name: '@acme/mdxforge-charts',
  version: '1.0.0',
  components: {
    MetricCard,
    RevenueChart
  },
  styles: ['./style.css'],
  rules: ['./rules.mdx'],
  componentRules: [
    {
      name: 'RevenueChart',
      description: 'Render a static revenue chart.',
      props: {
        data: {
          type: 'Array<{ month: string; value: number }>',
          required: true
        }
      },
      examples: [
        `<RevenueChart data={[{ month: "Jan", value: 12000 }]} />`
      ]
    }
  ]
})
```

The runtime type can stay deliberately small:

```ts
export interface MDXForgeExtension {
  name: string
  version: string
  components?: Record<string, React.ComponentType<unknown>>
  styles?: string[]
  rules?: string[]
  componentRules?: ComponentRule[]
}

export interface ComponentRule {
  name: string
  description: string
  props?: Record<string, ComponentPropRule>
  examples: string[]
}

export interface ComponentPropRule {
  type: string
  required?: boolean
  description?: string
  default?: string
}
```

## Installation Sources

The installer should support local code first, then remote packages. Publishing to npm or appearing in a marketplace must be optional.

```bash
mdxforge dev ./docs
mdxforge install ./packages/mdxforge-charts
mdxforge install npm:@acme/mdxforge-charts
mdxforge install git:https://github.com/acme/mdxforge-charts
```

The first-class sources should be:

| Source | Example | Purpose |
| --- | --- | --- |
| workspace local | `./docs/.mdxforge/extensions/charts` | Let a user write project-only components. |
| local folder | `./packages/mdxforge-charts` | Let a user install an unpacked extension. |
| npm | `npm:@acme/mdxforge-charts` | Let users distribute versioned packages. |
| git | `git:https://github.com/acme/mdxforge-charts` | Let users share source packages before npm. |

Future sources can include:

- zip or tarball URLs.
- a first-party package catalog.

Installed shared packages should live under the application data directory. Workspace-local packages should remain in the opened document workspace and should not require installation into a global package store.

```txt
userData/
  packages/
    @acme/mdxforge-charts/
      package.json
      dist/
      rules.mdx
      preview.png
  package-lock.json
  package-registry.json
```

Workspace-local package references should be declared by the workspace:

```json
{
  "packages": [
    "./.mdxforge/extensions/internal-components",
    "npm:@acme/mdxforge-charts"
  ]
}
```

The important product rule is:

```txt
local component code
-> local extension registration
-> MDX can use the registered component
```

No registry upload should be required for this flow.

## Local Development Mode

Local development mode is the path for users who want to write components directly on disk and immediately use them in local MDX files.

Recommended workspace layout:

```txt
docs/
  mdxforge.config.json
  index.mdx
  reports/
    revenue.mdx
  .mdxforge/
    extensions/
      local-dashboard/
        package.json
        src/
          extension.ts
          components/
            MetricCard.tsx
            RevenueChart.tsx
        rules.mdx
        dist/
          extension.js
          style.css
```

Workspace config:

```json
{
  "packages": ["./.mdxforge/extensions/local-dashboard"]
}
```

Local extension metadata:

```json
{
  "name": "local-dashboard",
  "private": true,
  "version": "0.0.0",
  "mdxforge": {
    "schema": 1,
    "type": "component",
    "entry": "./dist/extension.js",
    "source": "./src/extension.ts",
    "styles": ["./dist/style.css"],
    "rules": ["./rules.mdx"]
  },
  "dependencies": {
    "@mdxforge/extension": "workspace:*"
  }
}
```

Source entry:

```tsx
import { defineMdxForgeExtension } from '@mdxforge/extension'
import { MetricCard } from './components/MetricCard'
import { RevenueChart } from './components/RevenueChart'

export default defineMdxForgeExtension({
  name: 'local-dashboard',
  version: '0.0.0',
  components: {
    MetricCard,
    RevenueChart
  },
  componentRules: [
    {
      name: 'MetricCard',
      description: 'Display one dashboard metric.',
      props: {
        label: { type: 'string', required: true },
        value: { type: 'string', required: true }
      },
      examples: ['<MetricCard label="MRR" value="$24,800" />']
    }
  ]
})
```

MDX usage:

```mdx
---
title: Revenue Dashboard
description: A local MDX document using a workspace component.
---

# Revenue Dashboard

<MetricCard label="MRR" value="$24,800" />
```

Development loop:

<Steps>
  <Step>
    User creates a workspace extension folder under `.mdxforge/extensions`.
  </Step>
  <Step>
    MDXForge discovers it from `mdxforge.config.json`.
  </Step>
  <Step>
    MDXForge builds `src/extension.ts` to `dist/extension.js` or runs the package's own build command.
  </Step>
  <Step>
    MDXForge watches extension source files and reloads the component bundle when they change.
  </Step>
  <Step>
    The open MDX preview re-renders with the updated local component.
  </Step>
</Steps>

There are two possible build strategies:

| Strategy | Behavior | Best For |
| --- | --- | --- |
| package build script | Extension defines `scripts.mdxforgeBuild`, MDXForge runs it. | Complex local packages with custom tooling. |
| built-in builder | MDXForge bundles `source` with esbuild/vite and externalizes React. | Simple TSX components and fast iteration. |

The built-in builder should be the default for local development because it lets users write one or two TSX files without creating a full npm package pipeline.

```json
{
  "scripts": {
    "mdxforge:build": "vite build --config vite.extension.config.ts",
    "mdxforge:watch": "vite build --watch --config vite.extension.config.ts"
  }
}
```

If scripts exist, MDXForge can delegate to them. If not, MDXForge can use the `mdxforge.source` field and produce `dist/extension.js` itself.

The user-facing UI should make this feel local:

- "Add local component folder" in settings.
- "Create local extension" from an open workspace.
- "Rebuild local extensions" action.
- Package diagnostics showing build output and load errors.
- Quick link to copy merged AI rules for the current workspace.

## Marketplace Discovery

A public catalog can start as a static index generated from package metadata.

Discovery criteria:

- npm package has keyword `mdxforge-package`.
- package has a valid `mdxforge` field.
- package exposes a compatible package type.
- package declares preview metadata.

Catalog entry:

```json
{
  "name": "@acme/mdxforge-charts",
  "version": "1.0.0",
  "description": "Chart and metric components for MDXForge documents.",
  "types": ["component", "rules"],
  "source": "npm",
  "install": "npm:@acme/mdxforge-charts",
  "image": "https://cdn.example.com/previews/charts.png",
  "repository": "https://github.com/acme/mdxforge-charts"
}
```

The in-app marketplace can be read-only at first:

<Steps>
  <Step>
    Fetch a signed static catalog JSON from a known URL.
  </Step>
  <Step>
    Show package name, description, preview image, source, version, and permissions.
  </Step>
  <Step>
    Install by delegating to the same package installer used by the CLI.
  </Step>
  <Step>
    Let users enable or disable installed packages per application or per workspace.
  </Step>
</Steps>

## Build Pipeline

Extension authors should publish compiled browser-compatible bundles. MDXForge can later provide a builder, but the runtime should not rely on compiling arbitrary TSX during document render.

Recommended author workflow:

```bash
pnpm create mdxforge-extension charts
cd charts
pnpm build
pnpm publish
```

Generated package:

```txt
mdxforge-charts/
  src/
    extension.ts
    components/
      MetricCard.tsx
  dist/
    extension.js
    style.css
  rules.mdx
  demo.mdx
  preview.png
  package.json
```

The extension bundle should be ESM and externalize React:

```ts
export default {
  name: '@acme/mdxforge-charts',
  version: '1.0.0',
  components: {
    MetricCard,
    RevenueChart
  }
}
```

MDXForge should load the compiled bundle, not the source files.

## Runtime Loading

The current renderer calls:

```ts
<Mdx components={mdxComponents} />
```

Extensions fit naturally by merging component maps:

```ts
const mdxComponents = getMDXComponents(extensionComponents)
```

The main process should own package discovery and trust decisions. The renderer should receive a manifest and a safe list of bundle URLs.

```txt
main process
  read installed package registry
  validate mdxforge metadata
  resolve extension assets
  expose safe local URLs

renderer
  import extension bundle URLs
  collect components
  merge into getMDXComponents()
  render MDX
```

For local files, use a dedicated protocol instead of raw filesystem paths:

```txt
mdxforge-extension://@acme/mdxforge-charts/extension.js
mdxforge-extension://@acme/mdxforge-charts/style.css
```

This lets MDXForge restrict asset access to installed package directories.

## Security Model

Component packages are still executable JavaScript. The system should treat that honestly.

The product should have explicit safety modes instead of a single global on/off switch.

| Mode | Behavior | Best For |
| --- | --- | --- |
| Safe Preview | Render only MDXForge core components. Third-party packages are disabled. | Opening unknown documents or downloaded workspaces. |
| Trusted Packages | Load only packages the user has explicitly trusted. | Normal daily use. |
| Local Dev Mode | Build and load workspace-local source packages with file watching. | User-authored local components. |
| High-Permission Mode | Enable transform/resource packages after an extra trust prompt. | Advanced extensions. |

Safe Preview should be available from the title bar, package warning UI, and error screens. It gives users a quick way to read a document without running third-party package code.

Default component extensions should have these limits:

- Run in the sandboxed renderer only.
- No Node integration.
- No direct `fs`, `shell`, or process access.
- No preload API access beyond normal document rendering.
- No automatic network permission if a stricter future network policy exists.
- No compile-time hooks unless the package type is `transform`.

High-permission packages should be separated:

| Capability | Risk | Requirement |
| --- | --- | --- |
| React component render | Medium | User installs and enables package. |
| CSS injection | Medium | Show package styles in permission summary. |
| remark/rehype transform | High | Explicit trust prompt. |
| local resource resolver | High | Explicit trust prompt and path scope. |
| Node command execution | Very high | Not supported for normal MDXForge packages. |

Trust should be recorded at the package version or package integrity level, not only by package name. A package update can change code, so updating should require revalidation when the integrity changes.

```json
{
  "trustedPackages": [
    {
      "name": "@acme/mdxforge-charts",
      "version": "1.0.0",
      "integrity": "sha256-...",
      "trustedAt": "2026-05-23T00:00:00.000Z",
      "scope": "global"
    }
  ]
}
```

Workspace-local packages are different. They are user-authored code, so they can be easier to trust during development, but MDXForge should still make activation explicit:

```txt
This workspace defines local component code:

./.mdxforge/extensions/local-dashboard

Enable Local Dev Mode for this workspace?
```

Remote packages should default to disabled after install until the user clicks "Trust and enable".

Install UI should say what the package can do:

```txt
@acme/mdxforge-charts

Permissions:
- Registers 2 MDX components
- Loads 1 stylesheet
- Adds AI writing rules

This package runs JavaScript in the document preview renderer.
Install only packages you trust.
```

Unknown workspaces should open in Safe Preview by default when they declare packages that are not already trusted. The user can then choose:

- keep Safe Preview
- trust packages for this workspace
- trust packages globally
- inspect package source
- disable package references

This keeps "open a downloaded MDX folder" safer than "install and run every package it requests".

## Package Trust Flow

The package activation flow should be separate from the install flow.

<Steps>
  <Step>
    Discover packages from global settings and `mdxforge.config.json`.
  </Step>
  <Step>
    Resolve metadata and compute integrity for local or installed package code.
  </Step>
  <Step>
    Compare package identity, version, integrity, source, and permissions with the trust store.
  </Step>
  <Step>
    If trusted, load the package. If not trusted, open the document in Safe Preview and show a package warning.
  </Step>
  <Step>
    Let the user trust the package for this workspace or globally.
  </Step>
</Steps>

Package identity should include:

| Field | Reason |
| --- | --- |
| name | Human-readable package identity. |
| version | Updates change risk. |
| source | npm, git, local folder, or workspace local. |
| integrity | Detects changed code. |
| package type | Determines permissions. |
| requested capabilities | Explains what the package can do. |

The UI should avoid vague warnings. A useful prompt says exactly what will run:

```txt
This document wants to enable @acme/mdxforge-charts.

Source: npm
Version: 1.0.0
Permissions:
- Runs React components in the preview renderer
- Injects 1 stylesheet
- Adds AI writing rules

It cannot access Node, fs, shell, or the MDXForge preload API.
```

## Isolation Options

There are three realistic levels of isolation.

| Isolation | Description | Trade-Off |
| --- | --- | --- |
| Same renderer, sandboxed | Dynamic import component bundles into the current renderer. | Simple and fast, but third-party code shares the React page runtime. |
| Isolated iframe | Render MDX content in an iframe with a narrow message bridge. | Stronger isolation, harder layout, theming, TOC, and event integration. |
| Separate BrowserView/WebContents | Render documents in a separate Electron webContents. | Strongest Electron-level isolation, much more complex UI integration. |

Recommended first cut:

```txt
Safe Preview + sandboxed renderer + explicit trust store
```

Recommended long-term path:

```txt
untrusted package preview
-> isolated iframe or separate webContents
-> narrow bridge for theme, size, links, and diagnostics
```

The first version should not pretend that same-renderer package loading is perfectly safe. It should present it as trusted-code execution with reduced privileges.

## Workspace Activation

Users need control over where extensions apply.

Global activation:

```json
{
  "enabledPackages": ["@acme/mdxforge-charts"]
}
```

Workspace activation:

```json
{
  "packages": ["@acme/mdxforge-charts"],
  "disabledPackages": ["@old/unsafe-package"]
}
```

Recommended behavior:

- Global packages are available everywhere.
- Workspace packages are available only for that folder.
- If a document uses an unknown component, show which package might provide it when known.
- If a package fails to load, show a package error but keep the rest of the document usable when possible.

## AI Rule Aggregation

Extension packages are valuable only if AI can use them correctly. MDXForge should expose a generated authoring guide that merges:

- core MDXForge rules
- enabled package rules
- component prop schemas
- package examples

Example generated output:

```md
# Active MDXForge Writing Rules

## Core Rules

- Do not write import/export.
- Use only registered components.

## Installed Components

### MetricCard

Display one KPI with a label, value, and optional trend.

Props:
- label: string, required
- value: string, required
- trend: string

Example:

<MetricCard label="MRR" value="$24,800" trend="+12%" />
```

This generated guide can power:

- a “Copy AI writing rules” button
- a CLI command
- a file in the workspace
- future agent integrations

```bash
mdxforge rules ./docs > MDXFORGE_RULES.md
```

## Error Handling

The extension system needs clear errors because package failures will be common.

Useful error categories:

| Error | User Message |
| --- | --- |
| Unknown component | Component is not registered by any enabled package. |
| Package not installed | Workspace requires a package that is missing. |
| Package disabled | Package is installed but disabled for this workspace. |
| Bundle load failed | Package entry failed to load. |
| Component render failed | Component threw during rendering. |
| Style failed | Stylesheet failed to load. |
| Incompatible version | Package requires a newer MDXForge version. |

The preview should isolate failures:

- An MDX compile error should not crash the app.
- A component render error should show a block-level error when possible.
- A package load error should show in a package diagnostics panel.
- The user should have a quick action to disable the failing package.

## Versioning

Packages should declare compatibility:

```json
{
  "peerDependencies": {
    "mdxforge": ">=1.2.0 <2.0.0",
    "react": "^19.0.0"
  }
}
```

MDXForge should enforce:

- package metadata schema version
- minimum MDXForge version
- supported package types
- React compatibility when possible

Metadata can include an explicit schema version:

```json
{
  "mdxforge": {
    "schema": 1,
    "type": "component",
    "entry": "./dist/extension.js"
  }
}
```

## Suggested Implementation Phases

### Phase 0: Internal Extension Registry

Refactor current built-in components behind an internal registry:

```ts
registerBuiltinExtension({
  name: '@mdxforge/core',
  components: getCoreComponents(),
  rules: []
})
```

This reduces future migration risk without changing product behavior.

### Phase 1: Workspace Local Component Packages

Support packages loaded from the current workspace only. This phase proves that users can write components locally without publishing anything.

Acceptance criteria:

- Read `mdxforge.config.json` from the opened folder.
- Resolve local package paths relative to the workspace root.
- Read `package.json` `mdxforge` metadata.
- Build `mdxforge.source` with a built-in builder when no compiled entry exists.
- Load compiled ESM entry from the local workspace package.
- Watch local extension source files and reload preview after rebuild.
- Merge package components into MDX rendering.
- Load package CSS.
- Show local packages, build status, and load errors in settings.
- Generate merged AI writing rules for the current workspace.

### Phase 2: Local Package Installer

Add commands for local folders and workspace package management:

```bash
mdxforge create-extension ./docs local-dashboard
mdxforge install ./local-extension
mdxforge list
mdxforge enable @acme/mdxforge-charts
mdxforge disable @acme/mdxforge-charts
mdxforge rules ./docs
```

Acceptance criteria:

- Copy or link local packages into `userData/packages`.
- Add workspace-local package references to `mdxforge.config.json`.
- Maintain a package registry file.
- Validate metadata before enabling.
- Produce actionable install/load errors.

### Phase 3: npm and Git Sources

Add npm and git installers. This phase is for sharing beyond one machine; it should not be required for local component development.

Acceptance criteria:

- Install `npm:` package into app package storage.
- Install `git:` source by cloning and building or using published artifacts.
- Record source, version, integrity, and installed time.
- Support package updates.

### Phase 4: Marketplace Catalog

Add an in-app package browser.

Acceptance criteria:

- Fetch static catalog JSON.
- Render package cards with preview images.
- Show permissions before install.
- Install selected packages.
- Allow rating/review links later without blocking the first version.

### Phase 5: High-Permission Extensions

Add transform/resource packages only after the low-permission path is stable.

Acceptance criteria:

- Separate package type and permission prompts.
- Restrict local resource scopes.
- Log transform failures with file context.
- Let users disable high-permission packages per workspace.

## Open Design Decisions

These choices should be made before implementation starts:

1. Whether installed packages are global-only first or workspace-local first.
2. Whether extension bundles are loaded by dynamic `import()` or through an iframe/webview isolation boundary.
3. Whether package CSS is global, scoped by convention, or injected into a shadow root.
4. Whether the first marketplace is npm-keyword based or a curated first-party catalog.
5. Whether package authors must use an official builder or can publish any compatible ESM bundle.

## Recommended First Cut

Start small:

```txt
local package folder
-> package.json metadata
-> compiled ESM extension entry
-> renderer-only component registration
-> CSS loading
-> merged AI writing rules
```

This proves the product value without committing to npm installs, registry moderation, high-permission transforms, or a full marketplace.

After that works, npm/git installation and a catalog become distribution problems rather than architecture risks.
