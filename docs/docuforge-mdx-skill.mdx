# Docuforge MDX Skill

Use this skill when writing documentation for Docuforge.

Docuforge documents should be written in MDX by default. MDX is preferred because it is easier to read than HTML, more expressive than plain Markdown, and constrained by a predictable component whitelist.

## Output Contract

When asked to write Docuforge documentation, output complete `.mdx` content.

Requirements:

1. Include frontmatter.
2. Use normal Markdown for most content.
3. Use only supported Docuforge components.
4. Do not use `import` or `export`.
5. Do not define custom React components.
6. Do not use arbitrary JavaScript expressions.
7. Put code examples in fenced code blocks.
8. Prefer plain Markdown if unsure whether a component exists.

## Frontmatter

Every document should start with:

```mdx
---
title: Document Title
description: One sentence description.
---
```

Rules:

- `title` is required.
- `description` is recommended.
- Keep frontmatter simple.
- Do not put nested objects in frontmatter unless explicitly requested.

## Supported Markdown

Allowed:

```mdx
# Heading 1

## Heading 2

Paragraph text.

- Item A
- Item B

> Quote

[Link](https://example.com)

![Image](./image.png)

| A | B |
| --- | --- |
| 1 | 2 |
```

## Code Blocks

Use fenced code blocks.

````mdx
```ts
export function hello(name: string) {
  return `Hello ${name}`
}
```
````

Rules:

- Always set a language.
- Code blocks are display-only.
- Do not rely on code blocks executing.

## Official Supported Components

These components are supported by Docuforge today.

### Callout

Use for important notes, warnings, tips, and constraints.

```mdx
<Callout title="Note" type="info">
This is an important note.
</Callout>
```

Common `type` values:

- `info`
- `warn`
- `error`

Rules:

- Keep callouts short.
- Do not nest large sections inside callouts.
- Prefer `Callout` over bold paragraphs for warnings or key notes.

### Cards / Card

Use for resource links or option groups.

```mdx
<Cards>
  <Card title="Quick Start" href="https://example.com">
    Read the quick start guide.
  </Card>
  <Card title="API Reference" href="https://example.com/api">
    Browse API details.
  </Card>
</Cards>
```

Rules:

- `Card` must include `title`.
- `href` is optional.
- Use `Card` inside `Cards`.
- Keep card body concise.

### CodeBlockTabs / CodeBlockTab

Use when showing equivalent commands or snippets.

```mdx
<CodeBlockTabs defaultValue="pnpm">
  <CodeBlockTabsList>
    <CodeBlockTabsTrigger value="pnpm">pnpm</CodeBlockTabsTrigger>
    <CodeBlockTabsTrigger value="npm">npm</CodeBlockTabsTrigger>
  </CodeBlockTabsList>
  <CodeBlockTab value="pnpm">

```bash
pnpm install
```

  </CodeBlockTab>
  <CodeBlockTab value="npm">

```bash
npm install
```

  </CodeBlockTab>
</CodeBlockTabs>
```

Rules:

- `CodeBlockTab` must include `value`.
- `CodeBlockTabsTrigger` must include matching `value`.
- Prefer one code block per tab.

## Supported Base Elements

Docuforge also supports customized rendering for normal Markdown elements:

- `a`
- `img`
- `h1`
- `h2`
- `h3`
- `h4`
- `h5`
- `h6`
- `table`
- `pre`

Do not write these as JSX unless necessary. Prefer normal Markdown syntax.

## Temporary Compatibility Components

These are accepted by the renderer but currently display as fallback placeholders unless project-specific implementations are added later:

- `Accordion`
- `Accordions`
- `Step`
- `Steps`
- `File`
- `Files`
- `Tabs`
- `Tab`

Use them only when explicitly requested. Prefer normal Markdown otherwise.

Example:

```mdx
<Steps>
  <Step>
    Open the MDX file.
  </Step>
  <Step>
    Preview the rendered document.
  </Step>
</Steps>
```

## Forbidden Patterns

Do not write:

```mdx
import Component from './Component'

export const metadata = {}

function CustomComponent() {
  return <div />
}

<UnknownComponent />

{fetch('/api')}

{window.location.href}
```

## Decision Rules

- If the user asks for a note, warning, or limitation, use `Callout`.
- If the user asks for links/resources/options, use `Cards` and `Card`.
- If the user asks for multiple install methods or language variants, use `CodeBlockTabs`.
- If a component is not in the supported list, do not use it.
- If unsure, use plain Markdown.

## Default Template

```mdx
---
title: Title
description: One sentence summary.
---

# Title

Intro paragraph.

<Callout title="Note" type="info">
Important context.
</Callout>

## Section

Content goes here.

```ts
console.log('example')
```

## Resources

<Cards>
  <Card title="Resource" href="https://example.com">
    Resource description.
  </Card>
</Cards>
```
