# MDXForge 标签页设计判断

MDXForge 应保留标签页，但它不应该变成完整编辑器的标签页系统。

更准确的定位是：

> 标签页 = 多文档审阅时的阅读上下文栏。

<Callout title="结论" type="info">
保留标签页。  
但只做轻量多文档阅读，不做 VS Code 式复杂编辑器标签。
</Callout>

## 为什么需要标签页

MDXForge 的核心场景不是单篇 Markdown 阅读，而是：

1. AI 生成一组结构化 MDX / Markdown 文档
2. 用户本地打开文件夹
3. 在多个文档之间审阅、对照、导航
4. 最后修改、确认、分享或分发

这个流程天然会出现多文档切换。

例如：

- `README.mdx`
- `implementation-plan.mdx`
- `api-reference.mdx`
- `review-checklist.mdx`
- `changelog.md`

如果没有标签页，用户每次都只能依赖左侧文件树重新定位。  
这会让“对照阅读”变慢。

## 标签页真正解决的问题

标签页不只是视觉元素。它解决的是：

| 问题 | 标签页价值 |
| --- | --- |
| 多文档来回切换 | 保留最近阅读上下文 |
| AI 生成多份交付物 | 快速审阅多个文件 |
| 文件夹工作区 | 当前打开文档更清晰 |
| 右侧 Outline / AI 面板 | 明确 active document |
| 用户误点侧边栏 | 不必立刻丢失当前文档 |

## 不应该做什么

MDXForge 不是 IDE，也不是通用 Markdown 编辑器。

所以标签页首版不应包含这些能力：

- 拖拽排序
- 分屏编辑
- 多 tab group
- dirty state 管理
- session restore 复杂状态
- unsaved changes 弹窗
- 每个 tab 独立编辑器状态
- 类 VS Code 的完整标签页系统

<Callout title="边界" type="warn">
标签页服务阅读与审阅，不服务复杂编辑。
</Callout>

## 推荐规则

### 单文件模式

单文件打开时，标签页可以隐藏。

规则：

- 只有一个文件 → 不显示 tab bar
- 当前文件标题可放在 Workbench 顶部或文档头部
- 点击同一文件不创建新 tab

理由：  
单文件阅读时，标签页没有明显价值，只会占空间。

### 文件夹模式

文件夹工作区中保留标签页。

规则：

- 点击侧边栏文件 → 打开或激活 tab
- 已打开文件 → 直接激活
- 关闭当前 tab → 回到上一个 active tab
- tab 只保存 `path`、`title`、`lastActiveAt`
- 不保存复杂渲染状态


## 产品定义

可以在产品文档中这样描述：

> MDXForge 的标签页用于在工作区内快速切换已打开的文档。它是阅读与审阅上下文，不是完整编辑器工作区。

## 最终判断

保留。

但功能边界要明确：

- 要多文档阅读
- 不要复杂编辑器
- 要轻量状态
- 不要重型 session
- 要服务 AI 文档审阅
- 不要偏向通用 IDE

一句话：

> MDXForge 不是编辑器，但它是多文档审阅工具；多文档审阅需要轻量标签页。
