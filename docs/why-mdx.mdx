# 为什么使用 MDX

在 AI 时代，文档不只是给人看的，也越来越多地给 AI 看。

我们写知识库、产品文档、技术方案、组件说明，本质上都在做一件事：把复杂信息结构化，让人和 AI 都能快速理解。

Markdown 很轻量，但表达能力有限。HTML 表达能力很强，但太啰嗦，token 成本高。MDX 刚好站在中间：既保持文本的可读性，又允许加入组件化表达。

这也是我们选择做本地 MDX 预览工具的原因。

## Markdown 的问题：简单，但表达力不够

Markdown 最大优点是简单。

```md
# 标题

这是一段文字。

- 列表
- 列表
```

它很适合写文章、笔记、README。

但当文档变复杂时，Markdown 就开始吃力了。

比如你想表达：

- 提示框
- 步骤流
- Tabs
- 文件树
- API 参数表
- 代码组
- 可折叠区域
- 组件预览
- 交互式示例

纯 Markdown 很难优雅表达。

最后通常会变成：

```md
> 注意：这里是提示

### 步骤 1

### 步骤 2

#### Tab A

#### Tab B
```

能写，但结构不够明确。人看久了累，AI 解析也容易混。

Markdown 的问题不是不好，而是它太“平”。

## HTML 的问题：能表达，但太浪费

HTML 表达能力当然强。

```html
<div class="callout warning">
  <div class="title">注意</div>
  <div class="content">
    这里是一段提示内容。
  </div>
</div>
```

但问题是：太啰嗦。

同样一个提示框，用 HTML 会产生大量标签、class、嵌套结构。

对于 AI 来说，这些都是 token。

而很多 token 并没有真正传递知识，只是在描述 UI 外壳。

AI 读 HTML 时，常常要穿过一堆无意义结构，才能看到真正内容。

这会带来几个问题：

1. token 成本高
2. 上下文浪费
3. 可读性差
4. 修改成本高
5. 人类不愿意直接写

HTML 适合浏览器渲染，不适合人类长期手写，也不适合作为 AI-native 知识格式。

## MDX 的优势：表达力和 token 成本的平衡

MDX 可以理解为：

> Markdown + JSX 组件

它保留 Markdown 的文本可读性，同时允许插入组件。

例如：

```mdx
<Callout type="warning" title="注意">
  修改配置前，请先备份。
</Callout>
```

相比 HTML，它更短。相比 Markdown，它更明确。

AI 一看就知道：

- 这是一个 Callout
- 类型是 warning
- 标题是 注意
- 内容是 修改配置前，请先备份

没有多余 div。没有复杂 class。没有无意义嵌套。

这就是 MDX 的价值：用更少 token 表达更清楚的结构。

## MDX 比 Markdown 更可读

Markdown 的可读性来自“纯文本”。

但复杂文档里，纯文本不一定更清楚。

例如 Tabs：

```md
### React

React 示例内容。

### Vue

Vue 示例内容。
```

这对人来说勉强能懂，但结构其实不明确。

换成 MDX：

```mdx
<Tabs items={["React", "Vue"]}>
  <Tab value="React">
    React 示例内容。
  </Tab>

  <Tab value="Vue">
    Vue 示例内容。
  </Tab>
</Tabs>
```

虽然多了组件，但语义更清楚。

它明确告诉读者和 AI：这里不是两个普通标题，而是一组 Tabs。

所以 MDX 不是牺牲可读性，而是提升复杂内容的结构可读性。

## MDX 比 HTML 更省 token

HTML 最大问题是结构噪音太多。

例如一个步骤组件，如果用 HTML 写，可能是：

```html
<div class="steps">
  <div class="step">
    <div class="step-title">安装依赖</div>
    <div class="step-content">
      执行 pnpm install。
    </div>
  </div>
</div>
```

MDX 可以写成：

```mdx
<Steps>
  <Step title="安装依赖">
    执行 pnpm install。
  </Step>
</Steps>
```

更短，更清楚，更稳定。

对于 AI，上下文窗口很宝贵。我们不应该把 token 浪费在重复标签和样式类名上。

MDX 把 UI 结构压缩成组件语义。这就是它比 HTML 更适合 AI 的原因。

## 为什么需要本地 MDX 预览工具

MDX 很适合写文档，但也有一个问题：需要渲染环境。

普通 Markdown 编辑器打不开复杂 MDX 组件。很多在线工具又依赖项目框架、构建流程、远程服务。

所以我们需要一个本地工具：

- 打开 `.mdx` 文件
- 直接渲染
- 打开文件夹
- 切换文档
- 支持常见文档组件
- 接近官网文档体验
- 离线可用
- 不绑定 Next.js 项目

它的目标不是替代完整文档站，而是降低 MDX 的使用门槛。

让 MDX 像 Markdown 一样容易打开，像文档站一样好看。

## 面向 AI 的文档格式，应当具备三个特点

第一，**人类可读**。不能只有机器能看，人也要能直接编辑。

第二，**结构明确**。AI 不只是读文字，也需要理解内容层级、组件语义、信息关系。

第三，**token 经济**。同样的信息，表达越短，AI 能处理的上下文越多。

Markdown 满足第一点，但结构表达弱。HTML 满足第二点，但 token 太浪费。MDX 同时兼顾三点。

这就是它适合 AI-native 文档的原因。

## 结论

我们选择 MDX，不是因为它更酷，而是因为它更实用。

它比 Markdown 表达力更强。它比 HTML token 更省。它比纯文档更接近真实产品界面。它比复杂文档站更适合本地写作和预览。

在 AI 时代，文档格式不只是排版问题，而是知识表达效率问题。

MDX 的核心价值就是：

> 用接近 Markdown 的成本，获得接近组件系统的表达力。

这也是我们做这个本地 MDX 预览工具的原因。
